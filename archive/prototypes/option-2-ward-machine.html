Prototype 2 · Electoral · Castle ward, Leicester

Local knowledge, without locals

What it is: give it a target ward; it builds the electoral brief, finds the most useful local issue, and writes the doorstep script. How it uses an agent factory: head office feeds a target list in at one end; an orchestrator fans each ward out to electoral-analysis, research, drafting, and review agents running in parallel in the cloud; locally-flavoured campaigns come out the other, each held for one human's approval. This page walks the six rungs from a chat answer to a 120-ward battleground.

About this concept

User & problem

Jo, a party field organiser (party unnamed) assigned a target ward with no local members. The factory substitutes for local knowledge — which is exactly the capability and exactly the threat. Built on fully verified real terrain: Castle ward's 2023 results, the Castle–Knighton boundary fault line, live traffic orders.

Where the agents multiply — and why it isn't theatre

The electoral analyst, scout, and frames agents do separable read-heavy work in parallel (context isolation); the reviewer is deliberately a separate agent so invented social proof gets caught by fresh eyes (generator/reviewer separation). The factory rung adds queue, cloud sandboxes, evaluator checks (including banned-tactic rules), the approve/edit/reject/respond inbox, and per-run traces.

Risks & tensions

Ethically hottest of the four: a real school's safety scheme instrumentalised for electoral gain, real incumbents named. Party stays unnamed; staged as “look straight at what's coming”, never a pitch. Requires Ed + Hannah's explicit sign-off. Volunteer displacement is the native fear — no bolting on required.

Build complexity & 10-minute fit

Medium-high: prototype 1's spine plus the electoral layer (all verified open). Highest stage impact and panel fit of the four. UI mirrors shadcn (Awake template tokens); production would use shadcn/ui React components directly.
